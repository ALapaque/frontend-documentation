Traduire une application ne se résume pas à remplacer des chaînes. L'ordre des mots change, les pluriels suivent des règles par langue, et le texte concaténé devient intraduisible. Angular répond avec une i18n **au build** : tu marques le texte dans tes templates, un outil l'extrait, les traducteurs remplissent un fichier par langue, et Angular produit **un bundle par locale**. C'est le choix qui maximise la performance — aucun coût i18n au runtime — au prix d'un build par langue.

## Marquer le texte : l'attribut i18n

Tu n'enveloppes pas le texte dans une fonction ; tu poses l'attribut `i18n` sur l'élément qui le contient. Angular le repère à la compilation.

```html
<h1 i18n="page d'accueil|Titre principal@@home.title">
  Bienvenue
</h1>

<!-- Pour un attribut (title, placeholder, aria-label) : i18n-<attribut> -->
<img [src]="logo" i18n-alt alt="Logo de l'entreprise" />
```

La valeur `i18n` encode trois choses séparées par `|` et `@@` : `signification|description@@identifiant`. La **signification** regroupe les occurrences identiques d'un même sens (un seul « OK » traduit partout), la **description** aide le traducteur, et l'**identifiant** stable (`@@home.title`) évite qu'un simple changement de texte source casse toutes les traductions existantes.

:::callout{type="warn"}
**Ne concatène jamais des fragments traduits.** `"Tu as " + count + " messages"` est intraduisible : en allemand le verbe part à la fin, en français l'accord change. Marque la **phrase entière** comme une seule unité i18n et délègue les variables à ICU (ci-dessous). Un fragment isolé n'a pas de grammaire.
:::

## Variables et pluriels : les expressions ICU

Le piège `count === 1 ? "message" : "messages"` est faux dès qu'on quitte le français. Les expressions ICU, intégrées au format Angular, encodent la règle de pluriel de **chaque** langue.

```html
<p i18n="@@inbox.count">
  {count, plural,
    =0 {Aucun message}
    one {Un message}
    other {# messages}
  }
</p>
```

`#` est remplacé par la valeur. La forme `select` gère les cas non numériques (genre, statut) :

```html
<span i18n>
  {genre, select, male {Il} female {Elle} other {Iel}} a répondu.
</span>
```

Les catégories (`one`, `few`, `many`, `other`) sont celles de CLDR : le traducteur arabe ou polonais en ajoute, sans que tu touches au code.

## Extraire et traduire

Une commande extrait toutes les unités marquées vers un fichier pivot.

```bash
ng extract-i18n --output-path src/locale
# produit src/locale/messages.xlf
```

Tu dupliques ce fichier par langue (`messages.fr.xlf`, `messages.de.xlf`) et les traducteurs remplissent les balises `<target>`. Le fichier source reste la **vérité** ; un identifiant manquant signale une traduction à faire.

## Construire : un bundle par locale

Tu déclares les locales et leurs traductions dans la configuration, puis Angular génère une sortie par langue.

```json
// angular.json (extrait)
"i18n": {
  "sourceLocale": "fr",
  "locales": { "de": "src/locale/messages.de.xlf" }
},
"build": { "options": { "localize": true } }
```

Le résultat : `dist/<projet>/fr/` et `dist/<projet>/de/`, chacun servi sous son préfixe d'URL. `LOCALE_ID` est figé dans chaque bundle ; les pipes `DatePipe`/`CurrencyPipe` s'alignent automatiquement sur la locale du build.

:::callout{type="info"}
Pense à enregistrer les données de locale pour les pipes hors anglais : Angular les inclut au build `localize`, mais en usage manuel (`formatDate`) tu importes et passes la locale. Le formatage lui-même reste l'affaire d'`Intl` sous le capot — voir le module web sur l'API `Intl`.
:::

## Build-time ou runtime : l'arbitrage

L'i18n officielle est **build-time** : performance maximale, zéro JavaScript de traduction côté client, mais changer de langue impose de charger l'autre bundle (un autre préfixe d'URL, donc un rechargement), et ajouter une langue impose un rebuild + redéploiement.

:::compare
::bad
```text
Besoin : l'utilisateur bascule FR ⇄ AR instantanément, sans rechargement,
et on ajoute des langues sans redéployer.
→ @angular/localize seul ne le permet pas (une langue = un build).
```
::
::good
```text
Solution runtime : Transloco (ou ngx-translate) charge des JSON de
traduction à la volée et change de langue sans rebuild.
→ léger coût runtime, mais bascule dynamique et ajout de langue à chaud.
```
::
:::

**Pourquoi.** Si ton besoin est « un site multilingue servi par langue », `@angular/localize` gagne : rien à charger, rien à exécuter. Si ton besoin est « bascule instantanée, langues ajoutées sans déploiement » (apps internes, white-label), une bibliothèque runtime comme Transloco est le bon outil. Le critère n'est pas la mode, c'est : la langue est-elle connue à la construction ou choisie en direct ?

## RTL : ce n'est pas qu'une traduction

Arabe, hébreu, persan s'écrivent de droite à gauche. La traduction ne suffit pas : la mise en page doit se miroiter.

```html
<html [attr.dir]="locale === 'ar' ? 'rtl' : 'ltr'">
```

Côté CSS, emploie les **propriétés logiques** (`margin-inline-start` plutôt que `margin-left`, `padding-inline-end`) : elles suivent `dir` automatiquement, là où `left`/`right` restent figées et cassent le miroir.
