React n'a pas d'i18n intégrée : tu choisis une bibliothèque. `react-intl` (la couche React de **FormatJS**) est la plus répandue parce qu'elle s'appuie sur le standard **ICU MessageFormat** et sur l'API `Intl` du navigateur. L'idée centrale : un message n'est pas une chaîne avec des `+`, c'est un **patron** qui connaît les pluriels, le genre et le formatage, et que chaque langue remplit à sa façon.

## Le provider et le premier message

`IntlProvider` injecte la locale et le catalogue de messages dans l'arbre. Les composants lisent ensuite par `id`.

```jsx
import { IntlProvider, FormattedMessage } from "react-intl";

<IntlProvider locale="fr" messages={messagesFr} defaultLocale="en">
  <App />
</IntlProvider>;

// Dans un composant :
<FormattedMessage id="home.title" defaultMessage="Bienvenue" />;
```

Le `defaultMessage` sert de repli et de source d'extraction (voir plus bas). Pour du texte hors JSX (un `placeholder`, un `aria-label`), passe par le hook :

```jsx
const intl = useIntl();
<input placeholder={intl.formatMessage({ id: "search.placeholder", defaultMessage: "Rechercher" })} />;
```

## ICU MessageFormat : pluriels, select, rich text

C'est le cœur. Le réflexe `count === 1 ? "article" : "articles"` casse hors français/anglais ; ICU encode la règle de chaque langue dans le message lui-même.

```jsx
<FormattedMessage
  id="cart.count"
  defaultMessage="{count, plural, =0 {Panier vide} one {# article} other {# articles}}"
  values={{ count }}
/>;
```

`#` reçoit la valeur. ICU gère aussi `select` (genre, statut) et le **rich text** — tu passes des balises comme valeurs, jamais en concaténant du JSX autour de fragments traduits :

```jsx
<FormattedMessage
  id="terms"
  defaultMessage="J'accepte les <a>conditions</a>."
  values={{ a: (chunks) => <a href="/cgu">{chunks}</a> }}
/>;
```

:::callout{type="warn"}
**Ne découpe jamais une phrase pour y glisser un lien ou un nombre par concaténation.** L'ordre des mots diffère d'une langue à l'autre : le lien peut passer en tête, le nombre changer l'accord. Garde la phrase entière dans `defaultMessage` et injecte les variables et les balises via `values`.
:::

## Extraire et compiler les messages

Plutôt que de maintenir les `id` à la main, FormatJS extrait les `defaultMessage` du code source.

```bash
formatjs extract "src/**/*.{js,jsx,ts,tsx}" --out-file lang/fr.json
```

Tu obtiens un catalogue `{ id: message }` que les traducteurs dupliquent par langue. En production, on **compile** ces JSON (en AST ou en chaînes) pour éviter de parser l'ICU au runtime — un gain net sur les gros catalogues.

## Charger les locales paresseusement

Embarquer toutes les langues dans le bundle initial est un gâchis : l'utilisateur n'en lit qu'une. Charge le catalogue à la demande via un `import()` dynamique, puis remonte-le dans le provider.

```jsx
function useMessages(locale) {
  const [messages, setMessages] = useState(null);
  useEffect(() => {
    let actif = true;
    import(`./lang/${locale}.json`).then((m) => actif && setMessages(m.default));
    return () => { actif = false; };
  }, [locale]);
  return messages;
}
```

Le bundler découpe alors chaque locale en un chunk séparé, téléchargé seulement quand on bascule dessus. C'est le même principe que le code splitting de route.

## RTL : miroiter, pas seulement traduire

Pour l'arabe ou l'hébreu, la traduction ne suffit pas : la mise en page se miroite.

```jsx
<html dir={locale === "ar" ? "rtl" : "ltr"} lang={locale} />;
```

En CSS, emploie les **propriétés logiques** (`margin-inline-start`, `padding-inline-end`, `inset-inline`) plutôt que `left`/`right` : elles suivent `dir` automatiquement. Les valeurs `start`/`end` de Flexbox et Grid s'orientent déjà toutes seules.
